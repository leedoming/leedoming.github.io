Face Swap Infrastructure Redesign

Real-world service · Production scaling & infra redesign

Fieldguide fit: Focused on production reliability, cost efficiency, and system constraints rather than model experimentation.
Fieldguide Connection → Identifying cost inefficiency in a production system and driving an end-to-end infrastructure redesign — including analysis, decision-making under failure, and knowledge transfer — directly reflects P&G's emphasis on using technology to improve efficiency and decision-making.
Business Impact

Reduced infrastructure cost by about 80% while improving concurrency handling under event traffic, replacing an always-on setup with a more practical on-demand architecture.

Before → After 4× m5.xlarge EC2 (24/7, ~$617/mo) → ECS Fargate on-demand (variable, ~$0 idle) · FastAPI async + tempfile isolation
📋 STAR — "Give an example of improving a process using technology."
S ~15%
A Face Swap service targeting nationwide university festivals was running 4× m5.xlarge EC2 instances 24/7, incurring a fixed cost of ~$617/month regardless of actual demand. The legacy codebase was unmaintainable, and concurrent requests caused processing errors — making multi-user support impossible during high-traffic festival periods.
T ~15%
Two junior engineers (including me) needed to analyze root causes, refactor the legacy code, reduce costs, and solve concurrent request handling — simultaneously. I owned the infrastructure pipeline and Docker environment configuration.
A ⭐ ~50%
Code refactor: Rewrote the service in FastAPI with per-request tempfile.NamedTemporaryFile isolation to solve concurrent collisions. Tuned ONNX Runtime session options (intra_op_num_threads, ORT_ENABLE_ALL) to maximize CPU inference performance.

Infrastructure decision-making (including a failure):
  • Implemented and validated on EC2 → shared with senior → received suggestion to use fully managed service
  • Attempted SageMaker Serverless first (ML-appropriate) → hit model size limits with InsightFace + GFPGAN multi-model parallel structure
  • Analyzed the constraint → pivoted to ECS Fargate on-demand as the correct fit
Forward compatibility: Pre-configured onnxruntime-gpu switchover and Dockerized the image so GPU upgrade for production was one config change.

Knowledge transfer: Taught a team member Docker from scratch — image/container concepts, Dockerfile authoring, ECR push, Fargate deploy — until they could work independently.
Champion Productivity — 80% cost reduction, fixed→variable Innovate for Growth — SageMaker failure → ECS pivot Execute with Excellence — code-level optimization Lead with Courage — junior-led, accepted senior input Bring Out Your Best — Docker training for teammate
R ~20%
Converted infrastructure from ~$617/month fixed to on-demand variable cost with $0 idle cost, achieving approximately 80% cost reduction during actual event operation. Resolved concurrent request issue entirely. Completed with GPU-ready Docker image and full team handoff.
4 instances → 1 (−75%) $617/mo → $0 idle ~80% cost reduction Concurrent requests solved GPU-ready handoff
Contributions
Redesigned deployment across three stages — EC2 → SageMaker Serverless → ECS Fargate — based on concrete system constraints.
Fixed concurrency bottlenecks by restructuring FastAPI file handling and request isolation.
Improved latency by 3+ seconds through parallelization and runtime tuning.
Eliminated idle compute costs by moving from always-on instances to on-demand container execution.

3-Stage Pipeline + DataOps YOLOv8 (detection) → CLIP (classification) → Claude 3.5 Sonnet (explanation) · SageMaker 5-step Pipeline · OpenSearch RAG + SNS alert · Auto-retraining loop
